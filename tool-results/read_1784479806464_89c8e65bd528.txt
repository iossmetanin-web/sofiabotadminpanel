    19→## Overview
    20→
    21→Hexagonal Architecture, also known as the Ports and Adapters pattern, was introduced by Alistair Cockburn in 2005. Its central thesis is that an application's core logic should be completely isolated from the outside world — no dependency on databases, web frameworks, message queues, or external APIs. The application communicates with the outside world exclusively through abstract interfaces called **ports**, which are implemented by concrete **adapters**.
    22→
    23→For a Telegram AI bot, this means your conversation logic, user management rules, and AI orchestration remain pure Python code with zero imports from aiogram, SQLAlchemy, Redis, or OpenAI. The bot handler is just an adapter that translates Telegram updates into calls on your application's ports. The database is just an adapter that persists what your domain dictates.
    24→
    25→The practical benefit is enormous: you can swap PostgreSQL for MongoDB, replace aiogram with a webhook-based API, or switch from OpenAI to a local model — all without touching a single line of business logic.

    29→## Core Concepts
    30→
    31→### Ports
    32→
    33→A port is an abstract interface that defines the contract between the application core and the outside world. Ports are defined in the application layer and express **what** the core needs or provides, never **how** those needs are satisfied.
    34→
    35→There are two kinds of ports:
    36→
    37→- **Inbound (driving) ports** — Interfaces that the outside world calls to interact with the application. These are your use case interfaces. The application core defines them and depends on them.
    38→- **Outbound (driven) ports** — Interfaces that the application core calls to interact with external systems. These are your repository interfaces, service interfaces, and gateway interfaces. The application core defines them and depends on them.
    39→
    40→In Python, ports are expressed as `typing.Protocol` classes or abstract base classes. The key constraint is that a port contains zero implementation details about any specific technology.
    41→
    42→```python
    43→# application/ports/user_repository.py
    44→from __future__ import annotations
    45→
    46→from typing import Protocol
    47→from uuid import UUID
    48→
    49→from domain.entities.user import User
    50→from domain.value_objects.telegram_id import TelegramId
    51→
    52→
    53→class UserRepository(Protocol):
    54→    """Outbound port: defines how the application persists and retrieves users."""
    55→
    56→    async def get_by_id(self, user_id: UUID) -> User | None:
    57→        """Retrieve a user by internal UUID. Returns None if not found."""
    58→        ...
    59→
    60→    async def get_by_telegram_id(self, telegram_id: TelegramId) -> User | None:
    61→        """Retrieve a user by their Telegram identifier. Returns None if not found."""
    62→        ...
    63→
    64→    async def save(self, user: User) -> User:
    65→        """Persist a user entity. Returns the saved entity with generated fields."""
    66→        ...
    67→
    68→    async def exists_by_telegram_id(self, telegram_id: TelegramId) -> bool:
    69→        """Check whether a user with the given Telegram ID already exists."""
    70→        ...
    71→```
    72→
    73→### Adapters
    74→
    75→An adapter is a concrete implementation of a port. Adapters live in the infrastructure or presentation layer and contain all the technology-specific code — SQL queries, HTTP calls, Redis commands, framework-specific types.
    76→
    77→Adapters translate between the outside world's representation and the domain's representation. A PostgreSQL adapter translates table rows into domain entities. An aiogram adapter translates `Message` objects into use case input DTOs.
    78→
    79→```python
    80→# infrastructure/database/repositories/user_repository.py
    81→from __future__ import annotations
    82→
    83→from uuid import UUID
    84→
    85→from sqlalchemy import select
    86→from sqlalchemy.ext.asyncio import AsyncSession
    87→
    88→from domain.entities.user import User
    89→from domain.value_objects.telegram_id import TelegramId
    90→from infrastructure.database.models.user import UserModel
    91→from infrastructure.database.mappers.user_mapper import UserMapper
    92→
    93→
    94→class PostgresUserRepository:
    95→    """Driven adapter: implements UserRepository using PostgreSQL via SQLAlchemy."""
    96→
    97→    def __init__(self, session: AsyncSession) -> None:
    98→        self._session = session
    99→        self._mapper = UserMapper()
   100→
   101→    async def get_by_id(self, user_id: UUID) -> User | None:
   102→        stmt = select(UserModel).where(UserModel.id == user_id)
   103→        result = await self._session.execute(stmt)
   104→        model = result.scalar_one_or_none()
   105→        if model is None:
   106→            return None
   107→        return self._mapper.to_entity(model)
   108→
   109→    async def get_by_telegram_id(self, telegram_id: TelegramId) -> User | None:
   110→        stmt = select(UserModel).where(UserModel.telegram_id == telegram_id.value)
   111→        result = await self._session.execute(stmt)
   112→        model = result.scalar_one_or_none()
   113→        if model is None:
   114→            return None
   115→        return self._mapper.to_entity(model)
   116→
   117→    async def save(self, user: User) -> User:
   118→        model = self._mapper.to_model(user)
   119→        self._session.add(model)
   120→        await self._session.flush()
   121→        return self._mapper.to_entity(model)
   122→
   123→    async def exists_by_telegram_id(self, telegram_id: TelegramId) -> bool:
   124→        stmt = select(UserModel.id).where(UserModel.telegram_id == telegram_id.value)
   125→        result = await self._session.execute(stmt)
   126→        return result.scalar_one_or_none() is not None
   127→```

   129→### Driving and Driven Sides
   130→
   131→The hexagon metaphor divides the world into two regions:
   132→
   133→```
   134→                    DRIVING SIDE
   135→                 (Left Side / Inbound)
   136→            ┌──────────────────────────┐
   137→            │   Telegram Updates       │
   138→            │   Admin Dashboard API    │
   139→            │   Scheduled Jobs         │
   140→            │   CLI Commands           │
   141→            └─────────┬────────────────┘
   142→                      │ calls
   143→                      ▼
   144→            ┌──────────────────────────┐
   145→            │                          │
   146→            │     APPLICATION CORE     │
   147→            │   (Domain + Use Cases)   │
   148→            │                          │
   149→            └─────────┬────────────────┘
   150→                      │ calls through
   151→                      ▼
   152→            ┌──────────────────────────┐
   153→            │   PostgreSQL             │
   154→            │   Redis                  │
   155→            │   OpenAI / Anthropic     │
   156→            │   Email / Webhooks       │
   157→            └──────────────────────────┘
   158→                 DRIVEN SIDE
   159→                (Right Side / Outbound)
   160→```
   161→
   162→**Driving adapters** (left side) initiate interaction with the application. They call inbound ports. In a Telegram bot, the primary driving adapter is the aiogram message handler that receives a Telegram `Update` and invokes a use case.
   163→
   164→**Driven adapters** (right side) are called by the application. They implement outbound ports. In a Telegram bot, driven adapters include database repositories, LLM provider clients, cache backends, and notification services.

   168→## Mapping Hexagonal to Aiogram Bot Structure
   169→
   170→The four-layer architecture from SKILL.md maps directly to hexagonal concepts:
   171→
   172→| Layer | Hexagonal Role | Contains |
   173→|-------|---------------|----------|
   174→| Domain | The hexagon core | Entities, value objects, domain events, domain exceptions |
   175→| Application | Port definitions + use cases | Inbound ports (use case interfaces), outbound ports (repository/service protocols), DTOs |
   176→| Infrastructure | Driven adapters | Repository implementations, LLM providers, Redis storage, cache backends |
   177→| Presentation | Driving adapters | Aiogram handlers, middleware, keyboards, filters |
   178→
   179→The key insight is that **ports are defined in the application layer**, not in infrastructure. Infrastructure implements those ports. This is what makes the dependency inversion work — the core defines what it needs, and the outer layers provide it.
   180→
   181→```
   182→bot/
   183→├── domain/                          # Hexagon core (innermost)
   184→│   ├── entities/
   185→│   ├── value_objects/
   186→│   ├── events/
   187→│   └── exceptions/
   188→├── application/                     # Port definitions + orchestration
   189→│   ├── use_cases/                   # Inbound port implementations
   190→│   ├── dto/                         # Data contracts crossing port boundaries
   191→│   ├── ports/                       # Outbound port interfaces (Protocol classes)
   192→│   │   ├── user_repository.py       # Protocol: how to persist users
   193→│   │   ├── chat_repository.py       # Protocol: how to persist chats
   194→│   │   ├── llm_provider.py          # Protocol: how to generate AI responses
   195→│   │   ├── cache_backend.py         # Protocol: how to cache data
   196→│   │   └── notification_service.py  # Protocol: how to send notifications
   197→│   └── services/
   198→├── infrastructure/                  # Driven adapters (implement outbound ports)
   199→│   ├── database/repositories/       # Implements repository protocols
   200→│   ├── llm/providers/               # Implements LLM provider protocol
   201→│   ├── redis/                       # Implements cache backend protocol
   202→│   └── logging/
   203→└── presentation/                    # Driving adapters (call inbound ports)
   204→    ├── handlers/                    # Translate Telegram updates → use case calls
   205→    ├── keyboards/
   206→    ├── filters/
   207→    ├── middleware/
   208→    └── fsm/
   209→```

   213→## Primary Ports — Driving Adapters
   214→
   215→Primary ports are the entry points into the application. In a Telegram bot, these are the use case classes that driving adapters call. Each use case represents a single application operation — a user registration, a message being sent, an AI response being generated.
   216→
   217→### Use Case as Inbound Port
   218→
   219→```python
   220→# application/use_cases/register_user.py
   221→from __future__ import annotations
   222→
   223→import structlog
   224→from uuid import UUID
   225→
   226→from application.dto.user_dto import RegisterUserRequest, RegisterUserResponse
   227→from application.ports.user_repository import UserRepository
   228→from application.ports.cache_backend import CacheBackend
   229→from domain.entities.user import User
   230→from domain.value_objects.telegram_id import TelegramId
   231→from domain.value_objects.language import Language
   232→from domain.exceptions.domain_exceptions import UserAlreadyExistsError
   233→
   234→logger = structlog.get_logger()
   235→
   236→
   237→class RegisterUser:
   238→    """Inbound port implementation: registers a new Telegram user.
   239→
   240→    This use case is called by the aiogram StartHandler driving adapter.
   241→    It orchestrates domain logic and delegates persistence to outbound ports.
   242→    """
   243→
   244→    def __init__(
   245→        self,
   246→        user_repository: UserRepository,
   247→        cache_backend: CacheBackend,
   248→    ) -> None:
   249→        self._user_repository = user_repository
   250→        self._cache_backend = cache_backend
   251→
   252→    async def execute(self, request: RegisterUserRequest) -> RegisterUserResponse:
   253→        telegram_id = TelegramId(request.telegram_id)
   254→        language = Language(request.language)
   255→
   256→        if await self._user_repository.exists_by_telegram_id(telegram_id):
   257→            raise UserAlreadyExistsError(
   258→                f"User with telegram_id={request.telegram_id} already exists"
   259→            )
   260→
   261→        user = User.create(
   262→            telegram_id=telegram_id,
   263→            username=request.username,
   264→            first_name=request.first_name,
   265→            language=language,
   266→        )
   267→
   268→        saved_user = await self._user_repository.save(user)
   269→
   270→        await self._cache_backend.set(
   271→            key=f"user:{saved_user.id}",
   272→            value=saved_user.to_dict(),
   273→            ttl=3600,
   274→        )
   275→
   276→        logger.info(
   277→            "user_registered",
   278→            user_id=str(saved_user.id),
   279→            telegram_id=request.telegram_id,
   280→        )
   281→
   282→        return RegisterUserResponse(
   283→            user_id=saved_user.id,
   284→            telegram_id=saved_user.telegram_id.value,
   285→            username=saved_user.username,
   286→            language=saved_user.language.code,
   287→        )
   288→```
   289→
   290→### Aiogram Handler as Driving Adapter
   291→
   292→The aiogram handler is a driving adapter — it translates the Telegram `Message` into a use case request and translates the use case response back into a Telegram reply.
   293→
   294→```python
   295→# presentation/handlers/user_handlers.py
   296→from __future__ import annotations
   297→
   298→from aiogram import Router
   299→from aiogram.types import Message
   300→from aiogram.filters import CommandStart
   301→import structlog
   302→
   303→from application.use_cases.register_user import RegisterUser
   304→from application.dto.user_dto import RegisterUserRequest
   305→from application.ports.user_repository import UserRepository
   306→from application.ports.cache_backend import CacheBackend
   307→from domain.exceptions.domain_exceptions import UserAlreadyExistsError
   308→from presentation.keyboards.main_keyboard import build_main_keyboard
   309→
   310→logger = structlog.get_logger()
   311→
   312→
   313→def create_user_router(
   314→    register_user_use_case: RegisterUser,
   315→) -> Router:
   316→    """Driving adapter factory: creates an aiogram Router for user handlers.
   317→
   318→    The router translates Telegram updates into use case invocations.
   319→    No business logic lives here — only input parsing, output formatting,
   320→    and error translation.
   321→    """
   322→    router = Router()
   323→
   324→    @router.message(CommandStart())
   325→    async def handle_start_command(message: Message) -> None:
   326→        """Driving adapter: translates /start into RegisterUser use case."""
   327→        request = RegisterUserRequest(
   328→            telegram_id=message.from_user.id,
   329→            username=message.from_user.username or "",
   330→            first_name=message.from_user.first_name or "",
   331→            language=message.from_user.language_code or "en",
   332→        )
   333→
   334→        try:
   335→            response = await register_user_use_case.execute(request)
   336→            keyboard = build_main_keyboard(response.language)
   337→            await message.answer(
   338→                f"Welcome, {response.first_name}! Your account is ready.",
   339→                reply_markup=keyboard,
   340→            )
   341→        except UserAlreadyExistsError:
   342→            keyboard = build_main_keyboard(request.language)
   343→            await message.answer(
   344→                "Welcome back! Your account already exists.",
   345→                reply_markup=keyboard,
   346→            )
   347→        except Exception:
   348→            logger.exception("start_command_failed", telegram_id=message.from_user.id)
   349→            await message.answer("Something went wrong. Please try again later.")
   350→
   351→    return router
   352→```

   356→## Secondary Ports — Driven Adapters
   357→
   358→Secondary ports are the outbound interfaces that the application core depends on. The application defines these as `Protocol` classes, and infrastructure adapters provide concrete implementations.
   359→
   360→### LLM Provider Port
   361→
   362→```python
   363→# application/ports/llm_provider.py
   364→from __future__ import annotations
   365→
   366→from typing import AsyncIterator, Protocol
   367→
   368→from application.dto.llm_dto import GenerateRequest, GenerateResponse
   369→
   370→
   371→class LLMProvider(Protocol):
   372→    """Outbound port: defines how the application generates AI responses.
   373→
   374→    Any LLM provider (OpenAI, Anthropic, local model) must implement
   375→    this interface. The application core never knows which provider
   376→    is actually being used.
   377→    """
   378→
   379→    async def generate(self, request: GenerateRequest) -> GenerateResponse:
   380→        """Generate a complete AI response for the given prompt."""
   381→        ...
   382→
   383→    async def stream(self, request: GenerateRequest) -> AsyncIterator[str]:
   384→        """Stream an AI response token by token. Yields text chunks."""
   385→        ...
   386→
   387→    async def count_tokens(self, text: str) -> int:
   388→        """Count the number of tokens in the given text for the active model."""
   389→        ...
   390→
   391→    @property
   392→    def model_name(self) -> str:
   393→        """Return the identifier of the model currently in use."""
   394→        ...
   395→
   396→    @property
   397→    def max_context_tokens(self) -> int:
   398→        """Return the maximum context window size for the active model."""
   399→        ...
   400→```

   437→## Adapter Implementations for Common Services
   438→
   439→### Redis Cache Adapter
   440→
   441→```python
   442→# infrastructure/redis/cache.py
   443→from __future__ import annotations
   444→
   445→import json
   446→from typing import Any
   447→
   448→import redis.asyncio as aioredis
   449→import structlog
   450→
   451→logger = structlog.get_logger()
   452→
   453→
   454→class RedisCacheAdapter:
   455→    """Driven adapter: implements CacheBackend using Redis."""
   456→
   457→    def __init__(self, redis_client: aioredis.Redis, key_prefix: str = "bot") -> None:
   458→        self._redis = redis_client
   459→        self._prefix = key_prefix
   460→
   461→    def _prefixed(self, key: str) -> str:
   462→        return f"{self._prefix}:{key}"
   463→
   464→    async def get(self, key: str) -> Any | None:
   465→        raw = await self._redis.get(self._prefixed(key))
   466→        if raw is None:
   467→            return None
   468→        return json.loads(raw)
   469→
   470→    async def set(self, key: str, value: Any, ttl: int | None = None) -> None:
   471→        serialized = json.dumps(value, default=str)
   472→        if ttl is not None:
   473→            await self._redis.setex(self._prefixed(key), ttl, serialized)
   474→        else:
   475→            await self._redis.set(self._prefixed(key), serialized)
   476→
   477→    async def delete(self, key: str) -> None:
   478→        await self._redis.delete(self._prefixed(key))
   479→
   480→    async def exists(self, key: str) -> bool:
   481→        return bool(await self._redis.exists(self._prefixed(key)))
   482→
   483→    async def increment(self, key: str, amount: int = 1) -> int:
   484→        result = await self._redis.incrby(self._prefixed(key), amount)
   485→        return result
   486→```

   488→### OpenAI Adapter
   489→
   490→```python
   491→# infrastructure/llm/providers/openai_provider.py
   492→from __future__ import annotations
   493→
   494→from typing import AsyncIterator
   495→
   496→import openai
   497→import structlog
   498→
   499→from application.dto.llm_dto import GenerateRequest, GenerateResponse
   500→
   501→logger = structlog.get_logger()
   502→
   503→
   504→class OpenAIProviderAdapter:
   505→    """Driven adapter: implements LLMProvider using the OpenAI API."""
   506→
   507→    def __init__(
   508→        self,
   509→        api_key: str,
   510→        model: str = "gpt-4o",
   511→        max_context_tokens: int = 128000,
   512→    ) -> None:
   513→        self._client = openai.AsyncOpenAI(api_key=api_key)
   514→        self._model = model
   515→        self._max_context_tokens = max_context_tokens
   516→
   517→    async def generate(self, request: GenerateRequest) -> GenerateResponse:
   518→        messages = self._build_messages(request)
   519→        response = await self._client.chat.completions.create(
   520→            model=self._model,
   521→            messages=messages,
   522→            temperature=request.temperature,
   523→            max_tokens=request.max_tokens,
   524→        )
   525→        choice = response.choices[0]
   526→        return GenerateResponse(
   527→            text=choice.message.content or "",
   528→            tokens_used=response.usage.total_tokens if response.usage else 0,
   529→            model=self._model,
   530→            finish_reason=choice.finish_reason or "stop",
   531→        )
   532→
   533→    async def stream(self, request: GenerateRequest) -> AsyncIterator[str]:
   534→        messages = self._build_messages(request)
   535→        stream = await self._client.chat.completions.create(
   536→            model=self._model,
   537→            messages=messages,
   538→            temperature=request.temperature,
   539→            max_tokens=request.max_tokens,
   540→            stream=True,
   541→        )
   542→        async for chunk in stream:
   543→            delta = chunk.choices[0].delta
   544→            if delta.content:
   545→                yield delta.content
   546→
   547→    async def count_tokens(self, text: str) -> int:
   548→        response = await self._client.chat.completions.create(
   549→            model=self._model,
   550→            messages=[{"role": "user", "content": text}],
   551→            max_tokens=1,
   552→        )
   553→        return response.usage.prompt_tokens if response.usage else 0
   554→
   555→    @property
   556→    def model_name(self) -> str:
   557→        return self._model
   558→
   559→    @property
   560→    def max_context_tokens(self) -> int:
   561→        return self._max_context_tokens
   562→
   563→    @staticmethod
   564→    def _build_messages(request: GenerateRequest) -> list[dict[str, str]]:
   565→        messages: list[dict[str, str]] = []
   566→        if request.system_prompt:
   567→            messages.append({"role": "system", "content": request.system_prompt})
   568→        for msg in request.conversation_history:
   569→            messages.append({"role": msg.role, "content": msg.content})
   570→        messages.append({"role": "user", "content": request.user_message})
   571→        return messages
   572→```

   576→## Testing Strategy with Hexagonal Architecture
   577→
   578→Hexagonal architecture enables a layered testing strategy where each test scope exercises a different combination of adapters. The key principle: **test with fakes, not mocks**. A fake adapter is a working in-memory implementation of a port. It behaves like the real adapter but without external dependencies.
   579→
   580→### Fake Repository for Application Tests
   581→
   582→```python
   583→# tests/application/fakes/fake_user_repository.py
   584→from __future__ import annotations
   585→
   586→from uuid import UUID
   587→
   588→from domain.entities.user import User
   589→from domain.value_objects.telegram_id import TelegramId
   590→
   591→
   592→class FakeUserRepository:
   593→    """Fake adapter: in-memory UserRepository for testing use cases."""
   594→

   749→## Putting It All Together
   750→
   751→The wiring in `main.py` connects driving adapters to use cases and driven adapters to outbound ports. This is the composition root — the only place where concrete classes are coupled together.
   752→
   753→```python
   754→# main.py — Composition Root
   755→from __future__ import annotations
   756→
   757→import redis.asyncio as aioredis
   758→from aiogram import Bot, Dispatcher
   759→
   760→from application.use_cases.register_user import RegisterUser
   761→from infrastructure.database.repositories.user_repository import PostgresUserRepository
   762→from infrastructure.llm.providers.openai_provider import OpenAIProviderAdapter
   763→from infrastructure.redis.cache import RedisCacheAdapter
   764→from presentation.handlers.user_handlers import create_user_router
   765→from config import Settings
   766→
   767→
   768→async def main() -> None:
   769→    settings = Settings()
   770→
   771→    # Driven adapters (outbound port implementations)
   772→    redis_client = aioredis.from_url(settings.redis_url)
   773→    cache_backend = RedisCacheAdapter(redis_client)
   774→    # user_repository is created per-request inside middleware
   775→
   776→    # Use cases (inbound port implementations)
   777→    register_user = RegisterUser(
   778→        user_repository=PostgresUserRepository(session),  # injected per-request
   779→        cache_backend=cache_backend,
   780→    )
   781→
   782→    # Driving adapters (call inbound ports)
   783→    user_router = create_user_router(register_user_use_case=register_user)
   784→
   785→    # Wire everything into the Dispatcher
   786→    bot = Bot(token=settings.bot_token)
   787→    dp = Dispatcher()
   788→    dp.include_router(user_router)
   789→
   790→    await dp.start_polling(bot)
   791→```
   792→
   793→This composition root is the only file that knows about both sides of the hexagon. Everything else depends on abstractions. Swap `RedisCacheAdapter` for `MemcachedCacheAdapter` by changing one line here — no other file is affected.
