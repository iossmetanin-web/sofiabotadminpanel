     1→     1→# Domain-Driven Design for Telegram Bots
     2→     2→
     3→     3→## Overview
     4→     4→
     5→     5→Domain-Driven Design (DDD), formulated by Eric Evans in 2003, provides a systematic approach to building software that reflects the mental models of domain experts. For Telegram bots, DDD offers a vocabulary and a set of tactical patterns that help manage the inherent complexity of conversational interfaces, AI integration, and multi-user interactions.
     6→     6→
     7→     7→A Telegram bot is fundamentally a domain-rich application. It mediates between human users, a messaging platform, and AI services. Each of these concerns carries its own language, rules, and invariants. DDD gives us the tools to model these concerns explicitly, keeping the business logic at the center of the design rather than treating it as an afterthought to framework mechanics.
     8→     8→
     9→     9→---
    10→    10→
    11→    11→## Bounded Contexts for Bot Applications
    12→    12→
    13→    13→A bounded context is a linguistic boundary within which a particular model applies consistently. Words mean the same thing everywhere within a context, but may mean something different in another context. In a Telegram bot, the word "message" means different things depending on which context you are in.
    14→    14→
    15→    15→### Identifying Bounded Contexts
    16→    16→
    17→    17→For a feature-rich Telegram AI bot, the following bounded contexts emerge naturally:
    18→    18→
    19→    19→#### 1. Chat Context
    20→    20→
    21→    21→The Chat context manages Telegram chat interactions — what happens in the messaging interface itself. It deals with incoming updates, outbound responses, keyboard navigation, and conversation flow.
    22→    22→
    23→    23→**Ubiquitous Language:**
    24→    24→- **Chat** — A Telegram chat (private, group, or channel)
    25→    25→- **Conversation** — A back-and-forth exchange between user and bot
    26→    26→- **Turn** — A single user message + bot response pair
    27→    27→- **Dialog** — A structured multi-step interaction (registration, settings)
    28→    28→- **Greeting** — The initial message when a user starts the bot
    29→    29→
    30→    30→#### 2. User Context
    31→    31→
    32→    32→The User context manages user identity, profiles, preferences, and authorization. It is distinct from Telegram's concept of a user because your bot may have its own notion of roles, subscriptions, and account states.
    33→    33→
    34→    34→**Ubiquitous Language:**
    35→    35→- **User** — A person who interacts with the bot
    36→    36→- **Profile** — The user's configurable attributes (language, timezone)
    37→    37→- **Role** — Authorization level (member, moderator, admin)
    38→    38→- **Session** — The user's current authentication state
    39→    39→- **Preference** — A user-configurable setting
    40→    40→
    41→    41→#### 3. AI Context
    42→    42→
    43→    43→The AI context encapsulates interaction with language models. It manages prompts, conversation history, token budgets, and response processing. This context exists because "conversation" in the AI sense (a sequence of messages sent to an LLM) is fundamentally different from "conversation" in the chat sense (a user chatting with the bot).
    44→    44→
    45→    45→**Ubiquitous Language:**
    46→    46→- **Prompt** — The instruction sent to the language model
    47→    47→- **Completion** — The response generated by the model
    48→    48→- **Token** — A unit of text measurement for billing and limits
    49→    49→- **Context Window** — The maximum token budget for a single LLM call
    50→    50→- **System Instruction** — The behavioral directive for the AI
    51→    51→- **Hallucination** — A factually incorrect AI response
    52→    52→
    53→    53→#### 4. Admin Context
    54→    54→
    55→    55→The Admin context handles bot management, statistics, user moderation, and system configuration. It has its own authorization model and its own view of data.
    56→    56→
    57→    57→**Ubiquitous Language:**
    58→    58→- **Dashboard** — The administrative interface
    59→    59→- **Metric** — A measurable system attribute
    60→    60→- **Broadcast** — A message sent to all users
    61→    61→- **Moderation Action** — Ban, mute, or warn a user
    62→    62→- **Configuration** — A system-level setting
    63→    63→
    64→    64→#### 5. Billing Context (if applicable)
    65→    65→
    66→    66→The Billing context manages subscriptions, usage quotas, and payment processing.
    67→    67→
    68→    68→**Ubiquitous Language:**
    69→    69→- **Subscription** — A user's access plan
    70→    70→- **Quota** — A resource limit (messages per day, tokens per month)
    71→    71→- **Top-Up** — Adding credits to an account
    72→    72→- **Invoice** — A record of payment
    73→    73→

   104→   104→## Aggregates, Entities, and Value Objects in Bot Context
   105→   105→
   106→   106→### Aggregates
   107→   107→
   108→   108→An aggregate is a cluster of domain objects treated as a single unit for data changes. Each aggregate has a root entity through which all modifications must pass. External references can only point to the aggregate root, never to internal entities.
   109→   109→
   110→   110→In a Telegram bot, aggregates enforce consistency boundaries that map naturally to user interactions:
   111→   111→
   112→   112→```python
   113→   113→# domain/aggregates/chat_aggregate.py
   114→   114→from dataclasses import dataclass, field
   115→   115→from datetime import datetime
   116→   116→from uuid import UUID, uuid4
   117→   117→
   118→   118→from domain.entities.chat_message import ChatMessage
   119→   119→from domain.entities.chat_settings import ChatSettings
   120→   120→from domain.value_objects.telegram_id import TelegramId
   121→   121→from domain.value_objects.message_text import MessageText
   122→   122→from domain.events.domain_events import ChatMuted, ChatBlocked, MessageAdded
   123→   123→from domain.exceptions.domain_exceptions import (
   124→   124→    ChatBlockedException,
   125→   125→    MessageLimitExceededException,
   126→   126→)
   127→   127→
   128→   128→
   129→   129→@dataclass
   130→   130→class ChatAggregate:
   131→   131→    """
   132→   132→    Aggregate root for a Telegram chat.
   133→   133→
   134→   134→    Consistency boundary: all modifications to a chat's messages
   135→   135→    and settings must go through this aggregate root.
   136→   136→    """
   137→   137→    id: UUID = field(default_factory=uuid4)
   138→   138→    telegram_id: int = 0
   139→   139→    chat_type: str = "private"
   140→   140→    is_active: bool = True
   141→   141→    settings: ChatSettings = field(default_factory=ChatSettings)
   142→   142→    messages: list[ChatMessage] = field(default_factory=list)
   143→   143→    created_at: datetime = field(default_factory=datetime.utcnow)
   144→   144→
   145→   145→    _pending_events: list = field(default_factory=list, repr=False)
   146→   146→
   147→   147→    def add_user_message(self, text: str, user_id: int) -> ChatMessage:
   148→   148→        """Add a user message to the chat, enforcing all invariants."""
   149→   149→        if not self.is_active:
   150→   150→            raise ChatBlockedException("Cannot add messages to an inactive chat")
   151→   151→
   152→   152→        if len(self.messages) >= self.settings.max_messages:
   153→   153→            raise MessageLimitExceededException(
   154→   154→                f"Chat message limit of {self.settings.max_messages} reached"
   155→   155→            )
   156→   156→
   157→   157→        message_text = MessageText(text)
   158→   158→        message = ChatMessage(
   159→   159→            chat_id=self.id,
   160→   160→            sender_id=user_id,
   161→   161→            text=message_text,
   162→   162→        )
   163→   163→        self.messages.append(message)
   164→   164→
   165→   165→        self._pending_events.append(
   166→   166→            MessageAdded(chat_id=self.id, message_id=message.id)
   167→   167→        )
   168→   168→        return message
   169→   169→
   170→   170→    def mute(self, moderator_id: int) -> None:
   171→   171→        if not self.is_active:
   172→   172→            raise ChatBlockedException("Cannot mute a blocked chat")
   173→   173→        self.is_active = False
   174→   174→        self._pending_events.append(
   175→   175→            ChatMuted(chat_id=self.id, moderator_id=moderator_id)
   176→   176→        )
   177→   177→
   178→   178→    def unmute(self) -> None:
   179→   179→        if self.is_active:
   180→   180→            raise ValueError("Chat is not muted")
   181→   181→        self.is_active = True
   182→   182→
   183→   183→    def block(self, admin_id: int) -> None:
   184→   184→        self.is_active = False
   185→   185→        self._pending_events.append(
   186→   186→            ChatBlocked(chat_id=self.id, admin_id=admin_id)
   187→   187→        )
   188→   188→
   189→   189→    def get_recent_messages(self, count: int = 10) -> list[ChatMessage]:
   190→   190→        return self.messages[-count:]
   191→   191→
   192→   192→    def clear_history(self) -> None:
   193→   193→        self.messages.clear()
   194→   194→
   195→   195→    def collect_events(self) -> list:
   196→   196→        events = self._pending_events[:]
   197→   197→        self._pending_events.clear()
   198→   198→        return events
   199→   199→```

   201→   201→### Entities
   202→   202→
   203→   203→Entities have distinct identities that persist over time. Even if two entities have identical attributes, they are different if their IDs differ.
   204→   204→
   205→   205→```python
   206→   206→# domain/entities/chat_message.py
   207→   207→from dataclasses import dataclass, field
   208→   208→from datetime import datetime
   209→   209→from uuid import UUID, uuid4
   210→   210→
   211→   211→from domain.value_objects.message_text import MessageText
   212→   212→
   213→   213→
   214→   214→@dataclass
   215→   215→class ChatMessage:
   216→   216→    """Entity: a single message within a chat aggregate."""
   217→   217→    id: UUID = field(default_factory=uuid4)
   218→   218→    chat_id: UUID = field(default=None)
   219→   219→    sender_id: int = 0
   220→   220→    text: MessageText = field(default=None)
   221→   221→    is_bot_message: bool = False
   222→   222→    created_at: datetime = field(default_factory=datetime.utcnow)
   223→   223→    edited_at: datetime | None = None
   224→   224→
   225→   225→    def edit(self, new_text: str) -> None:
   226→   226→        self.text = MessageText(new_text)
   227→   227→        self.edited_at = datetime.utcnow()
   228→   228→
   229→   229→    def mark_as_bot_response(self) -> None:
   230→   230→        self.is_bot_message = True
   231→   231→```
   232→   232→
   233→   233→```python
   234→   234→# domain/entities/chat_settings.py
   235→   235→from dataclasses import dataclass
   236→   236→
   237→   237→
   238→   238→@dataclass
   239→   239→class ChatSettings:
   240→   240→    """Entity within chat aggregate: configurable chat parameters."""
   241→   241→    language: str = "en"
   242→   242→    max_messages: int = 1000
   243→   243→    ai_model: str = "gpt-4o-mini"
   244→   244→    temperature: float = 0.7
   245→   245→    system_prompt: str = "You are a helpful assistant."
   246→   246→    enable_history: bool = True
   247→   247→    auto_end_timeout_minutes: int = 30
   248→   248→
   249→   249→    def update_temperature(self, new_temp: float) -> None:
   250→   250→        if not 0.0 <= new_temp <= 2.0:
   251→   251→            raise ValueError("Temperature must be between 0.0 and 2.0")
   252→   252→        self.temperature = new_temp
   253→   253→
   254→   254→    def update_system_prompt(self, prompt: str) -> None:
   255→   255→        if len(prompt) > 2000:
   256→   256→            raise ValueError("System prompt must be under 2000 characters")
   257→   257→        self.system_prompt = prompt
   258→   258→```

   260→   260→### Value Objects
   261→   261→
   262→   262→Value objects are immutable, compared by value, and enforce validation at creation. They are the building blocks of entities and aggregates.
   263→   263→
   264→   264→```python
   265→   265→# domain/value_objects/telegram_id.py
   266→   266→from dataclasses import dataclass
   267→   267→
   268→   268→
   269→   269→@dataclass(frozen=True)
   270→   270→class TelegramId:
   271→   271→    """Value object: a Telegram user or chat identifier."""
   272→   272→    value: int
   273→   273→
   274→   274→    def __post_init__(self) -> None:
   275→   275→        if not isinstance(self.value, int):
   276→   276→            raise TypeError("TelegramId must be an integer")
   277→   277→        if self.value <= 0:
   278→   278→            raise ValueError("TelegramId must be positive")
   279→   279→
   280→   280→    def __int__(self) -> int:
   281→   281→        return self.value
   282→   282→
   283→   283→    def __hash__(self) -> int:
   284→   284→        return hash(self.value)
   285→   285→
   286→   286→    def __eq__(self, other: object) -> bool:
   287→   287→        if not isinstance(other, TelegramId):
   288→   288→            return NotImplemented
   289→   289→        return self.value == other.value
   290→   290→```
   291→   291→
   292→   292→```python
   293→   293→# domain/value_objects/token_budget.py
   294→   294→from dataclasses import dataclass
   295→   295→
   296→   296→
   297→   297→@dataclass(frozen=True)
   298→   298→class TokenBudget:
   299→   299→    """Value object: token allocation for an LLM call."""
   300→   300→    max_tokens: int
   301→   301→    used_tokens: int = 0
   302→   302→
   303→   303→    def __post_init__(self) -> None:
   304→   304→        if self.max_tokens <= 0:
   305→   305→            raise ValueError("Max tokens must be positive")
   306→   306→        if self.used_tokens < 0:
   307→   307→            raise ValueError("Used tokens cannot be negative")
   308→   308→        if self.used_tokens > self.max_tokens:
   309→   309→            raise ValueError("Used tokens cannot exceed max tokens")
   310→   310→
   311→   311→    @property
   312→   312→    def remaining(self) -> int:
   313→   313→        return self.max_tokens - self.used_tokens
   314→   314→
   315→   315→    @property
   316→   316→    def utilization(self) -> float:
   317→   317→        return self.used_tokens / self.max_tokens if self.max_tokens > 0 else 0.0
   318→   318→
   319→   319→    def spend(self, tokens: int) -> "TokenBudget":
   320→   320→        """Return a new TokenBudget with tokens spent."""
   321→   321→        return TokenBudget(
   322→   322→            max_tokens=self.max_tokens,
   323→   323→            used_tokens=self.used_tokens + tokens,
   324→   324→        )
   325→   325→
   326→   326→    def is_exhausted(self) -> bool:
   327→   327→        return self.remaining <= 0
   328→   328→```

   332→   332→## Domain Events and Event-Driven Communication
   333→   333→
   334→   334→Domain events capture significant state changes within a bounded context. They enable loose coupling between contexts — one context publishes an event, and other contexts subscribe without the publisher knowing who is listening.
   335→   335→
   336→   336→### Event Definitions
   337→   337→
   338→   338→```python
   339→   339→# domain/events/domain_events.py
   340→   340→from dataclasses import dataclass, field
   341→   341→from datetime import datetime
   342→   342→from uuid import UUID
   343→   343→
   344→   344→
   345→   345→@dataclass
   346→   346→class DomainEvent:
   347→   347→    """Base class for all domain events."""
   348→   348→    occurred_at: datetime = field(default_factory=datetime.utcnow)
   349→   349→    event_id: UUID = field(default_factory=uuid4)
   350→   350→
   351→   351→
   352→   352→# --- User Context Events ---
   353→   353→
   354→   354→@dataclass
   355→   355→class UserRegistered(DomainEvent):
   356→   356→    telegram_id: int = 0
   357→   357→    username: str = ""
   358→   358→
   359→   359→
   360→   360→@dataclass
   361→   361→class UserRoleChanged(DomainEvent):
   362→   362→    user_id: UUID = field(default=None)
   363→   363→    old_role: str = ""
   364→   364→    new_role: str = ""
   365→   365→
   366→   366→
   367→   367→@dataclass
   368→   368→class UserDeactivated(DomainEvent):
   369→   369→    user_id: UUID = field(default=None)
   370→   370→    reason: str = ""
   371→   371→
   372→   372→
   373→   373→# --- Chat Context Events ---
   374→   374→
   375→   375→@dataclass
   376→   376→class ChatMuted(DomainEvent):
   377→   377→    chat_id: UUID = field(default=None)
   378→   378→    moderator_id: int = 0
   379→   379→
   380→   380→
   381→   381→@dataclass
   382→   382→class ChatBlocked(DomainEvent):
   383→   383→    chat_id: UUID = field(default=None)
   384→   384→    admin_id: int = 0
   385→   385→
   386→   386→
   387→   387→@dataclass
   388→   388→class MessageAdded(DomainEvent):
   389→   389→    chat_id: UUID = field(default=None)
   390→   390→    message_id: UUID = field(default=None)
   391→   391→
   392→   392→
   393→   393→# --- AI Context Events ---
   394→   394→
   395→   395→@dataclass
   396→   396→class AIResponseGenerated(DomainEvent):
   397→   397→    conversation_id: UUID = field(default=None)
   398→   398→    tokens_used: int = 0
   399→   399→    model: str = ""
   400→   400→    latency_ms: int = 0
   401→   401→
   402→   402→
   403→   403→@dataclass
   404→   404→class TokenBudgetExhausted(DomainEvent):
   405→   405→    user_id: UUID = field(default=None)
   406→   406→    conversation_id: UUID = field(default=None)
   407→   407→
   408→   408→
   409→   409→# --- Billing Context Events ---
   410→   410→
   411→   411→@dataclass
   412→   412→class SubscriptionActivated(DomainEvent):
   413→   413→    user_id: UUID = field(default=None)
   414→   414→    plan: str = ""
   415→   415→
   416→   416→
   417→   417→@dataclass
   418→   418→class QuotaReached(DomainEvent):
   419→   419→    user_id: UUID = field(default=None)
   420→   420→    quota_type: str = ""
   421→   421→    limit: int = 0
   422→   422→```
   423→   423→
   424→   424→### Event Handler Example
   425→   425→
   426→   426→```python
   427→   427→# application/event_handlers/token_budget_handler.py
   428→   428→from domain.events.domain_events import AIResponseGenerated, TokenBudgetExhausted
   429→   429→
   430→   430→
   431→   431→class TokenBudgetEventHandler:
   432→   432→    """Handles AI events in the Billing context."""
   433→   433→
   434→   434→    def __init__(self, quota_service) -> None:
   435→   435→        self._quota_service = quota_service
   436→   436→
   437→   437→    async def handle_ai_response(self, event: AIResponseGenerated) -> None:
   438→   438→        await self._quota_service.deduct_tokens(
   439→   439→            user_id=event.conversation_id,  # resolved to user
   440→   440→            tokens=event.tokens_used,
   441→   441→        )
   442→   442→
   443→   443→    async def handle_budget_exhausted(self, event: TokenBudgetExhausted) -> None:
   444→   444→        await self._quota_service.notify_user_limit_reached(
   445→   445→            user_id=event.user_id,
   446→   446→        )
   447→   447→```

   451→   451→## Ubiquitous Language for Telegram Bots
   452→   452→
   453→   453→The ubiquitous language is the shared vocabulary used by developers and domain experts. In a Telegram bot project, "domain experts" include bot users, community moderators, and the bot operator. Every term in the codebase should map to a term in the ubiquitous language.
   454→   454→
   455→   455→### Core Glossary
   456→   456→
   457→   457→| Term | Definition | Code Representation |
   458→   458→|------|-----------|-------------------|
   459→   459→| **Bot** | The automated Telegram account that receives and sends messages | `Bot` (infrastructure) |
   460→   460→| **User** | A person who interacts with the bot | `User` (entity) |
   461→   461→| **Chat** | A Telegram conversation (1:1 or group) | `ChatAggregate` (aggregate) |
   462→   462→| **Message** | A single text/media unit sent by user or bot | `ChatMessage` (entity) |
   463→   463→| **Command** | A message starting with `/` that triggers bot behavior | Command handler (controller) |
   464→   464→| **Conversation** | An AI interaction session with message history | `AIConversation` (aggregate) |
   465→   465→| **Turn** | One user message + one AI response | `ConversationTurn` (value object) |
   466→   466→| **Prompt** | The instruction set given to the AI | `SystemPrompt` (value object) |
   467→   467→| **Completion** | The AI-generated response | `AICompletion` (entity) |
   468→   468→| **Token** | A unit of text measurement for LLM APIs | `TokenBudget` (value object) |
   469→   469→| **Subscription** | A user's access plan with quotas | `Subscription` (entity) |
   470→   470→| **Quota** | A resource limit (messages, tokens, features) | `Quota` (value object) |
   471→   471→| **Moderation** | Administrative actions on users/chats | `ModerationAction` (entity) |
   472→   472→| **Broadcast** | A message sent to multiple users | `Broadcast` (entity) |
   473→   473→
   474→   474→### Language Consistency Rules
   475→   475→
   476→   476→1. **Never use synonyms.** If the ubiquitous language says "Conversation" for AI sessions, never call it "Session" or "Dialogue" in code.
   477→   477→2. **Distinguish similar terms.** A "Chat" (Telegram construct) is different from a "Conversation" (AI construct). Never use them interchangeably.
   478→   478→3. **Define terms once.** Each term has a single authoritative definition in the glossary. Code comments should reference the glossary.
   479→   479→4. **Evolve the language.** As the domain understanding deepens, update the glossary and refactor code to match. The code is the language.

   483→   483→## Context Mapping Between Bot Features
   484→   484→
   485→   485→Context mapping defines the relationships between bounded contexts. These relationships determine how contexts communicate, what they share, and where translation boundaries exist.
   486→   486→
   487→   487→### Relationship Types in a Bot Application
   488→   488→
   489→   489→#### User ↔ Chat: Customer-Supplier
   490→   490→
   491→   491→The User context provides user identity information that the Chat context consumes. User is the supplier; Chat is the customer. When a user is deactivated, the Chat context must react.
   492→   492→
   493→   493→```python
   494→   494→# Context mapping: User supplies identity to Chat
   495→   495→# The Chat context depends on a UserFacade (anti-corruption layer)
   496→   496→
   497→   497→from abc import ABC, abstractmethod
   498→   498→
   499→   499→
   500→   500→class UserFacade(ABC):
   501→   501→    """Anti-corruption layer: Chat context's view of User context."""
   502→   502→    @abstractmethod
   503→   503→    async def is_user_active(self, telegram_id: int) -> bool: ...
   504→   504→
   505→   505→    @abstractmethod
   506→   506→    async def get_user_role(self, telegram_id: int) -> str: ...
   507→   507→
   508→   508→
   509→   509→# Implementation in infrastructure translates between contexts
   510→   510→class UserFacadeImpl(UserFacade):
   511→   511→    def __init__(self, user_repository) -> None:
   512→   512→        self._user_repo = user_repository
   513→   513→
   514→   514→    async def is_user_active(self, telegram_id: int) -> bool:
   515→   515→        user = await self._user_repo.get_by_telegram_id(telegram_id)
   516→   516→        return user is not None and user.is_active
   517→   517→
   518→   518→    async def get_user_role(self, telegram_id: int) -> str:
   519→   519→        user = await self._user_repo.get_by_telegram_id(telegram_id)
   520→   520→        return user.role.value if user else "guest"
   521→   521→```
   522→   522→
   523→   523→#### Chat ↔ AI: Customer-Supplier with ACL
   524→   524→
   525→   525→The Chat context produces user messages that the AI context consumes. The AI context returns completions that the Chat context displays. An Anti-Corruption Layer prevents Chat's "message" concept from leaking into AI's "prompt" concept.
   526→   526→
   527→   527→```python
   528→   528→# Anti-Corruption Layer between Chat and AI contexts
   529→   529→
   530→   530→class ChatToAITranslator:
   531→   531→    """Translates between Chat context messages and AI context prompts."""
   532→   532→
   533→   533→    def message_to_prompt_input(self, chat_message) -> dict:
   534→   534→        return {
   535→   535→            "role": "user" if not chat_message.is_bot_message else "assistant",
   536→   536→            "content": chat_message.text.value,
   537→   537→        }
   538→   538→
   539→   539→    def completion_to_response(self, ai_completion) -> str:
   540→   540→        return ai_completion.text
   541→   541→```
   542→   542→
   543→   543→#### Admin → All Contexts: Published Language
   544→   544→
   545→   545→The Admin context communicates with all other contexts through a published language — well-defined event schemas that other contexts subscribe to.
   546→   546→
   547→   547→```python
   548→   548→# Admin publishes events that all contexts can subscribe to
   549→   549→@dataclass
   550→   550→class UserBannedByAdmin(DomainEvent):
   551→   551→    """Published language event from Admin context."""
   552→   552→    telegram_id: int = 0
   553→   553→    reason: str = ""
   554→   554→    banned_by: int = 0
   555→   555→    banned_at: datetime = field(default_factory=datetime.utcnow)
   556→   556→```

   559→   559→
   560→   560→## Repository Pattern in DDD
   561→   561→
   562→   562→In DDD, repositories provide the illusion of an in-memory collection of aggregates. They encapsulate the mechanics of persistence, allowing the domain layer to remain ignorant of storage technology.
   563→   563→
   564→   564→### Repository Interfaces
   565→   565→
   566→   566→Repositories are defined per aggregate root. You never have a repository for a non-root entity within an aggregate — you access it through the aggregate root.
   567→   567→
   568→   568→```python
   569→   569→# domain/repositories/chat_repository.py
   570→   570→from abc import ABC, abstractmethod
   571→   571→from uuid import UUID
   572→   572→
   573→   573→from domain.aggregates.chat_aggregate import ChatAggregate
   574→   574→
   575→   575→
   576→   576→class ChatRepository(ABC):
   577→   577→    """
   578→   578→    Repository for ChatAggregate roots.
   579→   579→
   580→   580→    Provides collection-like semantics for persisting
   581→   581→    and retrieving complete chat aggregates.
   582→   582→    """
   583→   583→
   584→   584→    @abstractmethod
   585→   585→    async def get_by_id(self, chat_id: UUID) -> ChatAggregate | None:
   586→   586→        """Retrieve a complete chat aggregate by its ID."""
   587→   587→        ...
   588→   588→
   589→   589→    @abstractmethod
   590→   590→    async def get_by_telegram_id(self, telegram_id: int) -> ChatAggregate | None:
   591→   591→        """Find a chat by its Telegram chat ID."""
   592→   592→        ...
   593→   593→
   594→   594→    @abstractmethod
   595→   595→    async def save(self, chat: ChatAggregate) -> None:
   596→   596→        """Persist the entire chat aggregate, including all child entities."""
   597→   597→        ...
   598→   598→
   599→   599→    @abstractmethod
   600→   600→    async def get_active_chats(self, offset: int, limit: int) -> list[ChatAggregate]:
   601→   601→        """List active chats with pagination."""
   602→   602→        ...
   603→   603→
   604→   604→    @abstractmethod
   605→   605→    async def count_active(self) -> int:
   606→   606→        """Count active chats."""
   607→   607→        ...
   608→   608→```
   609→   609→
   610→   610→### Repository Implementation
   611→   611→
   612→   612→```python
   613→   613→# infrastructure/persistence/sqlalchemy_chat_repository.py
   614→   614→from uuid import UUID
   615→   615→
   616→   616→from sqlalchemy.ext.asyncio import AsyncSession
   617→   617→from sqlalchemy import select
   618→   618→
   619→   619→from domain.aggregates.chat_aggregate import ChatAggregate
   620→   620→from domain.repositories.chat_repository import ChatRepository
   621→   621→from infrastructure.persistence.models import ChatModel, MessageModel
   622→   622→from domain.entities.chat_message import ChatMessage
   623→   623→from domain.value_objects.message_text import MessageText
   624→   624→
   625→   625→
   626→   626→class SQLAlchemyChatRepository(ChatRepository):
   627→   627→    """Concrete repository: persists ChatAggregate via SQLAlchemy."""
   628→   628→
   629→   629→    def __init__(self, session: AsyncSession) -> None:
   630→   630→        self._session = session
   631→   631→
   632→   632→    async def get_by_id(self, chat_id: UUID) -> ChatAggregate | None:
   633→   633→        result = await self._session.execute(
   634→   634→            select(ChatModel).where(ChatModel.id == chat_id)
   635→   635→        )
   636→   636→        chat_model = result.scalar_one_or_none()
   637→   637→        if chat_model is None:
   638→   638→            return None
   639→   639→        return await self._reconstitute(chat_model)
   640→   640→
   641→   641→    async def get_by_telegram_id(self, telegram_id: int) -> ChatAggregate | None:
   642→   642→        result = await self._session.execute(
   643→   643→            select(ChatModel).where(ChatModel.telegram_id == telegram_id)
   644→   644→        )
   645→   645→        chat_model = result.scalar_one_or_none()
   646→   646→        if chat_model is None:
   647→   647→            return None
   648→   648→        return await self._reconstitute(chat_model)
   649→   649→
   650→   650→    async def save(self, chat: ChatAggregate) -> None:
   651→   651→        existing = await self._session.execute(
   652→   652→            select(ChatModel).where(ChatModel.id == chat.id)
   653→   653→        )
   654→   654→        chat_model = existing.scalar_one_or_none()
   655→   655→
   656→   656→        if chat_model is None:
   657→   657→            chat_model = ChatModel(
   658→   658→                id=chat.id,
   659→   659→                telegram_id=chat.telegram_id,
   660→   660→                chat_type=chat.chat_type,
   661→   661→                is_active=chat.is_active,
   662→   662→                language=chat.settings.language,
   663→   663→                max_messages=chat.settings.max_messages,
   664→   664→            )
   665→   665→            self._session.add(chat_model)
   666→   666→        else:
   667→   667→            chat_model.is_active = chat.is_active
   668→   668→
   669→   669→        # Persist messages
   670→   670→        for message in chat.messages:
   671→   671→            msg_result = await self._session.execute(
   672→   672→                select(MessageModel).where(MessageModel.id == message.id)
   673→   673→            )
   674→   674→            msg_model = msg_result.scalar_one_or_none()
   675→   675→            if msg_model is None:
   676→   676→                msg_model = MessageModel(
   677→   677→                    id=message.id,
   678→   678→                    chat_id=chat.id,
   679→   679→                    sender_id=message.sender_id,
   680→   680→                    text=message.text.value,
   681→   681→                    is_bot_message=message.is_bot_message,
   682→   682→                    created_at=message.created_at,
   683→   683→                )
   684→   684→                self._session.add(msg_model)
   685→   685→
   686→   686→        await self._session.flush()
   687→   687→
   688→   688→        # Publish domain events
   689→   689→        for event in chat.collect_events():
   690→   690→            # Event publishing handled by Unit of Work
   691→   691→            pass
   692→   692→
   693→   693→    async def _reconstitute(self, chat_model: ChatModel) -> ChatAggregate:
   694→   694→        """Rebuild aggregate from persistence models."""
   695→   695→        from domain.entities.chat_settings import ChatSettings
   696→   696→
   697→   697→        messages_result = await self._session.execute(
   698→   698→            select(MessageModel)
   699→   699→            .where(MessageModel.chat_id == chat_model.id)
   700→   700→            .order_by(MessageModel.created_at)
   701→   701→        )
   702→   702→        message_models = messages_result.scalars().all()
   703→   703→
   704→   704→        messages = [
   705→   705→            ChatMessage(
   706→   706→                id=m.id,
   707→   707→                chat_id=m.chat_id,
   708→   708→                sender_id=m.sender_id,
   709→   709→                text=MessageText(m.text),
   710→   710→                is_bot_message=m.is_bot_message,
   711→   711→                created_at=m.created_at,
   712→   712→            )
   713→   713→            for m in message_models
   714→   714→        ]
   715→   715→
   716→   716→        return ChatAggregate(
   717→   717→            id=chat_model.id,
   718→   718→            telegram_id=chat_model.telegram_id,
   719→   719→            chat_type=chat_model.chat_type,
   720→   720→            is_active=chat_model.is_active,
   721→   721→            settings=ChatSettings(
   722→   722→                language=chat_model.language,
   723→   723→                max_messages=chat_model.max_messages,
   724→   724→            ),
   725→   725→            messages=messages,
   726→   726→            created_at=chat_model.created_at,
   727→   727→        )
   728→   728→```

   730→   730→### Unit of Work with Repositories
   731→   731→
   732→   732→```python
   733→   733→# application/unit_of_work.py
   734→   734→from abc import ABC, abstractmethod
   735→   735→from contextlib import asynccontextmanager
   736→   736→from typing import AsyncGenerator
   737→   737→
   738→   738→from domain.repositories.chat_repository import ChatRepository
   739→   739→from domain.repositories.user_repository import UserRepository
   740→   740→
   741→   741→
   742→   742→class UnitOfWork(ABC):
   743→   743→    """Abstract unit of work: coordinates repository operations."""
   744→   744→
   745→   745→    chats: ChatRepository
   746→   746→    users: UserRepository
   747→   747→
   748→   748→    async def __aenter__(self) -> "UnitOfWork":
   749→   749→        return self
   750→   750→
   751→   751→    async def __aexit__(self, exc_type, exc_val, exc_tb) -> None:
   752→   752→        if exc_type is not None:
   753→   753→            await self.rollback()
   754→   754→        else:
   755→   755→            await self.commit()
   756→   756→
   757→   757→    @abstractmethod
   758→   758→    async def commit(self) -> None: ...
   759→   759→
   760→   760→    @abstractmethod
   761→   761→    async def rollback(self) -> None: ...
   762→   762→
   763→   763→
   764→   764→class SQLAlchemyUnitOfWork(UnitOfWork):
   765→   765→    """Concrete UoW wrapping an async SQLAlchemy session."""
   766→   766→
   767→   767→    def __init__(self, session_factory) -> None:
   768→   768→        self._session_factory = session_factory
   769→   769→
   770→   770→    async def __aenter__(self) -> "UnitOfWork":
   771→   771→        self._session = self._session_factory()
   772→   772→        self.chats = SQLAlchemyChatRepository(self._session)
   773→   773→        self.users = SQLAlchemyUserRepository(self._session)
   774→   774→        return self
   775→   775→
   776→   776→    async def __aexit__(self, exc_type, exc_val, exc_tb) -> None:
   777→   777→        await super().__aexit__(exc_type, exc_val, exc_tb)
   778→   778→        await self._session.close()
   779→   779→
   780→   780→    async def commit(self) -> None:
   781→   781→        await self._session.commit()
   782→   782→
   783→   783→    async def rollback(self) -> None:
   784→   784→        await self._session.rollback()
   785→   785→```

   789→   789→## Common Pitfalls
   790→   790→
   791→   791→### Pitfall 1: Treating the Whole Bot as One Context
   792→   792→
   793→   793→A bot with 20 commands, AI integration, and admin features is not a single domain. Resist the urge to have one `models.py` with all entities. Identify bounded contexts and give each its own model.
   794→   794→
   795→   795→### Pitfall 2: Anemic Aggregates
   796→   796→
   797→   797→If your aggregate is just a bag of data with getters and setters, it is not enforcing invariants. Aggregates must encapsulate rules — when you call `chat.add_user_message()`, the aggregate should validate the message, check limits, and raise exceptions if invariants are violated.
   798→   798→
   799→   799→### Pitfall 3: Bypassing Aggregate Roots
   800→   800→
   801→   801→Accessing child entities directly, without going through the aggregate root, breaks consistency guarantees. If you need to modify a `ChatMessage`, you must do it through `ChatAggregate`, not by loading `ChatMessage` directly from a repository.
   802→   802→
   803→   803→### Pitfall 4: Confusing Value Objects with Entities
   804→   804→
   805→   805→If an object has identity (it matters which one it is, not just what its values are), it is an entity. If it is defined entirely by its attributes, it is a value object. A `TelegramId` is a value object — two `TelegramId(123)` instances are the same. A `User` is an entity — two users named "Alice" with different IDs are different people.
   806→   806→
   807→   807→### Pitfall 5: Giant Repositories
   808→   808→
   809→   809→A repository should serve one aggregate root. If your `ChatRepository` has methods for loading users, querying AI conversations, and fetching admin statistics, it has too many responsibilities. Split it into separate repositories per aggregate root.
   810→   810→

   812→   812→
   813→   813→## Best Practices
   814→   814→
   815→   815→1. **Start with the ubiquitous language.** Before writing any code, define the key terms of your bot's domain. Write them in a glossary that all team members agree on.
   816→   816→
   817→   817→2. **Model aggregates around consistency boundaries.** Ask: "What must be consistent at the end of a single transaction?" Everything that must be consistent together belongs in the same aggregate.
   818→   818→
   819→   819→3. **Keep aggregates small.** An aggregate should fit comfortably in memory. If loading a Chat aggregate means loading 10,000 messages, the aggregate is too large. Consider pagination or event sourcing.
   820→   820→
   821→   821→4. **Use domain events for cross-context communication.** Never call another context's repository directly. Publish an event and let the other context subscribe.
   822→   822→
   823→   823→5. **Define repository interfaces in the domain layer.** The domain layer declares what it needs; the infrastructure layer provides it. This keeps the domain independent.
   824→   824→
   825→   825→6. **Name use cases after domain operations.** `RegisterUser`, not `InsertUserRecord`. `StartConversation`, not `CreateConversationRow`. The code should speak the domain language.
   826→   826→
   827→   827→7. **Validate in value objects.** Push validation as close to the data as possible. A `MessageText` value object that validates length and content at construction time prevents invalid data from ever entering the system.
   828→   828→
   829→   829→8. **Test domain logic in isolation.** Entity and aggregate tests should not need a database, Redis, or any external system. Mock only the repository interfaces.
