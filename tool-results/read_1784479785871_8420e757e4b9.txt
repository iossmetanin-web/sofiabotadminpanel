     1тЖТ     1тЖТ# Clean Architecture for Telegram Bots
     2тЖТ     2тЖТ
     3тЖТ     3тЖТ## Overview
     4тЖТ     4тЖТ
     5тЖТ     5тЖТClean Architecture, introduced by Robert C. Martin (Uncle Bob), establishes a system of layered dependencies where business logic remains insulated from external concerns like databases, web frameworks, and third-party APIs. For Telegram bots built with aiogram 3.x, Clean Architecture provides a structured approach to separating bot-handling mechanics from domain logic, making the system testable, maintainable, and adaptable to changing requirements.
     6тЖТ     6тЖТ
     7тЖТ     7тЖТThe central insight is that your bot's business rules тАФ how conversations flow, what constitutes a valid command, how user permissions are evaluated тАФ should not depend on whether you use aiogram, python-telegram-bot, or any other framework. The Telegram Bot API is a delivery mechanism, not the core of your application.
     8тЖТ     8тЖТ
     9тЖТ     9тЖТ---
    10тЖТ    10тЖТ
    11тЖТ    11тЖТ## Core Principles
    12тЖТ    12тЖТ
    13тЖТ    13тЖТ### The Dependency Rule
    14тЖТ    14тЖТ
    15тЖТ    15тЖТThe Dependency Rule is the foundational law of Clean Architecture: **source code dependencies must point inward only**. An inner layer cannot know anything about an outer layer. Specifically:
    16тЖТ    16тЖТ
    17тЖТ    17тЖТ- **Entities** never import from use cases, adapters, or frameworks
    18тЖТ    18тЖТ- **Use Cases** never import from adapters or frameworks
    19тЖТ    19тЖТ- **Interface Adapters** never import from frameworks
    20тЖТ    20тЖТ- **Frameworks** depend on all inner layers
    21тЖТ    21тЖТ
    22тЖТ    22тЖТThis means if you delete the `aiogram` package from your project, only the outermost layer should fail to compile. Your entity classes, use case logic, and adapter interfaces should remain perfectly intact.
    23тЖТ    23тЖТ
    24тЖТ    24тЖТ```
    25тЖТ    25тЖТтФМтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФР
    26тЖТ    26тЖТтФВ                   Frameworks                         тФВ
    27тЖТ    27тЖТтФВ   (aiogram, FastAPI, SQLAlchemy, Redis, OpenAI)     тФВ
    28тЖТ    28тЖТтФВ  тФМтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФР  тФВ
    29тЖТ    29тЖТтФВ  тФВ             Interface Adapters                 тФВ  тФВ
    30тЖТ    30тЖТтФВ  тФВ  (Controllers, Presenters, Gateways, Repos)   тФВ  тФВ
    31тЖТ    31тЖТтФВ  тФВ  тФМтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФР   тФВ  тФВ
    32тЖТ    32тЖТтФВ  тФВ  тФВ          Application Business Rules     тФВ   тФВ  тФВ
    33тЖТ    33тЖТтФВ  тФВ  тФВ            (Use Cases)                  тФВ   тФВ  тФВ
    34тЖТ    34тЖТтФВ  тФВ  тФВ  тФМтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФР   тФВ   тФВ  тФВ

    37тЖТ    37тЖТтФВ  тФВ  тФВ  тФФтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФШ   тФВ   тФВ  тФВ

    40тЖТ    40тЖТтФФтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФШ

    44тЖТ    44тЖТ
    45тЖТ    45тЖТ### Entity Independence
    46тЖТ    46тЖТ
    47тЖТ    47тЖТEntities encapsulate enterprise-wide business rules. In a Telegram bot context, an entity could represent a User, a Conversation, a Subscription, or an AI Message. These entities must be framework-agnostic:
    48тЖТ    48тЖТ
    49тЖТ    49тЖТ```python
    50тЖТ    50тЖТ# domain/entities/user.py тАФ PURE, no framework imports
    51тЖТ    51тЖТfrom dataclasses import dataclass, field
    52тЖТ    52тЖТfrom datetime import datetime
    53тЖТ    53тЖТfrom enum import Enum
    54тЖТ    54тЖТ
    55тЖТ    55тЖТ
    56тЖТ    56тЖТclass UserRole(Enum):
    57тЖТ    57тЖТ    USER = "user"
    58тЖТ    58тЖТ    MODERATOR = "moderator"
    59тЖТ    59тЖТ    ADMIN = "admin"
    60тЖТ    60тЖТ
    61тЖТ    61тЖТ
    62тЖТ    62тЖТ@dataclass
    63тЖТ    63тЖТclass User:
    64тЖТ    64тЖТ    """Core user entity тАФ knows nothing about Telegram, databases, or LLMs."""
    65тЖТ    65тЖТ    id: int
    66тЖТ    66тЖТ    telegram_id: int
    67тЖТ    67тЖТ    username: str
    68тЖТ    68тЖТ    role: UserRole
    69тЖТ    69тЖТ    created_at: datetime = field(default_factory=datetime.utcnow)
    70тЖТ    70тЖТ    is_active: bool = True
    71тЖТ    71тЖТ
    72тЖТ    72тЖТ    def can_perform_admin_actions(self) -> bool:
    73тЖТ    73тЖТ        return self.role in (UserRole.ADMIN, UserRole.MODERATOR)
    74тЖТ    74тЖТ
    75тЖТ    75тЖТ    def deactivate(self) -> None:
    76тЖТ    76тЖТ        if not self.is_active:
    77тЖТ    77тЖТ            raise ValueError("User is already deactivated")
    78тЖТ    78тЖТ        self.is_active = False
    79тЖТ    79тЖТ
    80тЖТ    80тЖТ    def promote_to_admin(self, promoter_role: UserRole) -> None:
    81тЖТ    81тЖТ        if promoter_role != UserRole.ADMIN:
    82тЖТ    82тЖТ            raise PermissionError("Only admins can promote users")
    83тЖТ    83тЖТ        self.role = UserRole.ADMIN
    84тЖТ    84тЖТ```
    85тЖТ    85тЖТ
    86тЖТ    86тЖТNotice that `User` contains business rules (`can_perform_admin_actions`, `deactivate`, `promote_to_admin`) but has zero imports from aiogram, SQLAlchemy, or any external library. This is entity independence in practice.
    87тЖТ    87тЖТ
    88тЖТ    88тЖТ### Use Case Orchestration
    89тЖТ    89тЖТ
    90тЖТ    90тЖТUse cases (also called interactors) encode application-specific business rules. They orchestrate the flow of data between entities and outer layers, but they define their own interfaces (ports) for external communication:
    91тЖТ    91тЖТ
    92тЖТ    92тЖТ```python
    93тЖТ    93тЖТ# application/use_cases/register_user.py
    94тЖТ    94тЖТfrom abc import ABC, abstractmethod
    95тЖТ    95тЖТfrom dataclasses import dataclass
    96тЖТ    96тЖТ
    97тЖТ    97тЖТfrom domain.entities.user import User, UserRole
    98тЖТ    98тЖТ
    99тЖТ    99тЖТ
   100тЖТ   100тЖТ@dataclass
   101тЖТ   101тЖТclass RegisterUserRequest:
   102тЖТ   102тЖТ    """Input DTO тАФ carries data into the use case."""
   103тЖТ   103тЖТ    telegram_id: int
   104тЖТ   104тЖТ    username: str
   105тЖТ   105тЖТ    referred_by: int | None = None
   106тЖТ   106тЖТ
   107тЖТ   107тЖТ
   108тЖТ   108тЖТ@dataclass
   109тЖТ   109тЖТclass RegisterUserResponse:
   110тЖТ   110тЖТ    """Output DTO тАФ carries data out of the use case."""
   111тЖТ   111тЖТ    user_id: int
   112тЖТ   112тЖТ    telegram_id: int
   113тЖТ   113тЖТ    username: str
   114тЖТ   114тЖТ    role: str
   115тЖТ   115тЖТ    is_new_user: bool
   116тЖТ   116тЖТ
   117тЖТ   117тЖТ
   118тЖТ   118тЖТclass UserRepository(ABC):
   119тЖТ   119тЖТ    """Port defined by the use case тАФ implemented by outer layers."""
   120тЖТ   120тЖТ    @abstractmethod
   121тЖТ   121тЖТ    async def get_by_telegram_id(self, telegram_id: int) -> User | None: ...
   122тЖТ   122тЖТ
   123тЖТ   123тЖТ    @abstractmethod
   124тЖТ   124тЖТ    async def save(self, user: User) -> User: ...
   125тЖТ   125тЖТ
   126тЖТ   126тЖТ
   127тЖТ   127тЖТclass ReferralService(ABC):
   128тЖТ   128тЖТ    """Port for referral logic тАФ another boundary."""
   129тЖТ   129тЖТ    @abstractmethod
   130тЖТ   130тЖТ    async def process_referral(self, new_user_id: int, referrer_id: int) -> None: ...
   131тЖТ   131тЖТ
   132тЖТ   132тЖТ
   133тЖТ   133тЖТclass RegisterUserUseCase:
   134тЖТ   134тЖТ    """Application use case тАФ orchestrates user registration."""
   135тЖТ   135тЖТ
   136тЖТ   136тЖТ    def __init__(
   137тЖТ   137тЖТ        self,
   138тЖТ   138тЖТ        user_repo: UserRepository,
   139тЖТ   139тЖТ        referral_service: ReferralService,
   140тЖТ   140тЖТ    ) -> None:
   141тЖТ   141тЖТ        self._user_repo = user_repo
   142тЖТ   142тЖТ        self._referral_service = referral_service
   143тЖТ   143тЖТ
   144тЖТ   144тЖТ    async def execute(self, request: RegisterUserRequest) -> RegisterUserResponse:
   145тЖТ   145тЖТ        existing = await self._user_repo.get_by_telegram_id(request.telegram_id)
   146тЖТ   146тЖТ        if existing is not None:
   147тЖТ   147тЖТ            return RegisterUserResponse(
   148тЖТ   148тЖТ                user_id=existing.id,
   149тЖТ   149тЖТ                telegram_id=existing.telegram_id,
   150тЖТ   150тЖТ                username=existing.username,
   151тЖТ   151тЖТ                role=existing.role.value,
   152тЖТ   152тЖТ                is_new_user=False,
   153тЖТ   153тЖТ            )
   154тЖТ   154тЖТ
   155тЖТ   155тЖТ        new_user = User(
   156тЖТ   156тЖТ            id=0,  # will be assigned by persistence
   157тЖТ   157тЖТ            telegram_id=request.telegram_id,
   158тЖТ   158тЖТ            username=request.username,
   159тЖТ   159тЖТ            role=UserRole.USER,
   160тЖТ   160тЖТ        )
   161тЖТ   161тЖТ        saved_user = await self._user_repo.save(new_user)
   162тЖТ   162тЖТ
   163тЖТ   163тЖТ        if request.referred_by is not None:
   164тЖТ   164тЖТ            await self._referral_service.process_referral(
   165тЖТ   165тЖТ                saved_user.id, request.referred_by
   166тЖТ   166тЖТ            )
   167тЖТ   167тЖТ
   168тЖТ   168тЖТ        return RegisterUserResponse(
   169тЖТ   169тЖТ            user_id=saved_user.id,
   170тЖТ   170тЖТ            telegram_id=saved_user.telegram_id,
   171тЖТ   171тЖТ            username=saved_user.username,
   172тЖТ   172тЖТ            role=saved_user.role.value,
   173тЖТ   173тЖТ            is_new_user=True,
   174тЖТ   174тЖТ        )
   175тЖТ   175тЖТ```
   176тЖТ   176тЖТ
   177тЖТ   177тЖТThe use case defines its own abstract interfaces (`UserRepository`, `ReferralService`). Outer layers implement these interfaces, never the reverse.

   181тЖТ   181тЖТ## Layer Mapping for Telegram Bots
   182тЖТ   182тЖТ
   183тЖТ   183тЖТThe four concentric layers of Clean Architecture map to concrete components in a Telegram bot project:
   184тЖТ   184тЖТ
   185тЖТ   185тЖТ### Layer 1: Entities (Enterprise Business Rules)
   186тЖТ   186тЖТ
   187тЖТ   187тЖТThe innermost layer contains domain entities, value objects, and domain events. These are pure Python classes with no external dependencies.
   188тЖТ   188тЖТ
   189тЖТ   189тЖТ| Component | Telegram Bot Examples |
   190тЖТ   190тЖТ|-----------|----------------------|
   191тЖТ   191тЖТ| Entities | `User`, `Chat`, `Message`, `Subscription`, `AIConversation` |
   192тЖТ   192тЖТ| Value Objects | `TelegramId`, `MessageText`, `Language`, `TokenCount` |
   193тЖТ   193тЖТ| Domain Events | `UserRegistered`, `MessageReceived`, `SubscriptionExpired` |
   194тЖТ   194тЖТ| Business Rules | Permission checks, validation logic, state transitions |
   195тЖТ   195тЖТ
   196тЖТ   196тЖТ### Layer 2: Use Cases (Application Business Rules)
   197тЖТ   197тЖТ
   198тЖТ   198тЖТUse cases encode the specific operations your bot performs. Each use case represents a single user intention.
   199тЖТ   199тЖТ
   200тЖТ   200тЖТ| Use Case | Description |
   201тЖТ   201тЖТ|----------|-------------|
   202тЖТ   202тЖТ| `RegisterUser` | First-time user onboarding |
   203тЖТ   203тЖТ| `StartConversation` | Initiate an AI chat session |
   204тЖТ   204тЖТ| `SendMessage` | Process and route a user message |
   205тЖТ   205тЖТ| `ManageSubscription` | Handle subscription lifecycle |
   206тЖТ   206тЖТ| `ExecuteAdminCommand` | Process admin-only operations |
   207тЖТ   207тЖТ
   208тЖТ   208тЖТ### Layer 3: Interface Adapters
   209тЖТ   209тЖТ
   210тЖТ   210тЖТThis layer translates data between the format convenient for use cases and the format convenient for external agencies.
   211тЖТ   211тЖТ
   212тЖТ   212тЖТ| Component | Telegram Bot Examples |
   213тЖТ   213тЖТ|-----------|----------------------|
   214тЖТ   214тЖТ| Controllers | aiogram message handlers, callback handlers |
   215тЖТ   215тЖТ| Presenters | Response formatters, keyboard builders |
   216тЖТ   216тЖТ| Gateways | Abstract repository implementations |
   217тЖТ   217тЖТ| DTOs | Request/Response data transfer objects |
   218тЖТ   218тЖТ
   219тЖТ   219тЖТ### Layer 4: Frameworks & Drivers
   220тЖТ   220тЖТ
   221тЖТ   221тЖТThe outermost layer contains frameworks, tools, and external services.
   222тЖТ   222тЖТ
   223тЖТ   223тЖТ| Component | Telegram Bot Examples |
   224тЖТ   224тЖТ|-----------|----------------------|
   225тЖТ   225тЖТ| Web Framework | aiogram Dispatcher, Router |
   226тЖТ   226тЖТ| Database | SQLAlchemy models, migrations |
   227тЖТ   227тЖТ| Cache | Redis client, FSM storage |
   228тЖТ   228тЖТ| LLM | OpenAI/Anthropic API clients |
   229тЖТ   229тЖТ| Message Queue | RabbitMQ, Redis Pub/Sub |
   230тЖТ   230тЖТ
   231тЖТ   231тЖТ---
   232тЖТ   232тЖТ
   233тЖТ   233тЖТ## How aiogram Fits into Clean Architecture
   234тЖТ   234тЖТ
   235тЖТ   235тЖТaiogram belongs squarely in the Frameworks layer. It is a delivery mechanism тАФ the way your application receives input and delivers output to users via Telegram. The critical discipline is keeping aiogram out of your use cases and entities.
   236тЖТ   236тЖТ
   237тЖТ   237тЖТ### The Adapter Pattern for aiogram
   238тЖТ   238тЖТ
   239тЖТ   239тЖТ```python
   240тЖТ   240тЖТ# adapters/telegram/user_controller.py
   241тЖТ   241тЖТfrom aiogram import Router
   242тЖТ   242тЖТfrom aiogram.types import Message
   243тЖТ   243тЖТfrom aiogram.filters import CommandStart
   244тЖТ   244тЖТ
   245тЖТ   245тЖТfrom application.use_cases.register_user import (
   246тЖТ   246тЖТ    RegisterUserUseCase,
   247тЖТ   247тЖТ    RegisterUserRequest,
   248тЖТ   248тЖТ)
   249тЖТ   249тЖТfrom adapters.telegram.presenters.user_presenter import UserPresenter
   250тЖТ   250тЖТ
   251тЖТ   251тЖТ
   252тЖТ   252тЖТrouter = Router()
   253тЖТ   253тЖТ
   254тЖТ   254тЖТ
   255тЖТ   255тЖТ@router.message(CommandStart())
   256тЖТ   256тЖТasync def handle_start(message: Message, register_user: RegisterUserUseCase) -> None:
   257тЖТ   257тЖТ    """Controller: translates aiogram Message into use case input."""
   258тЖТ   258тЖТ    request = RegisterUserRequest(
   259тЖТ   259тЖТ        telegram_id=message.from_user.id,
   260тЖТ   260тЖТ        username=message.from_user.username or "anonymous",
   261тЖТ   261тЖТ        referred_by=_extract_referral(message),
   262тЖТ   262тЖТ    )
   263тЖТ   263тЖТ    response = await register_user.execute(request)
   264тЖТ   264тЖТ
   265тЖТ   265тЖТ    presenter = UserPresenter()
   266тЖТ   266тЖТ    reply_text = presenter.format_welcome(response)
   267тЖТ   267тЖТ    await message.answer(reply_text)
   268тЖТ   268тЖТ
   269тЖТ   269тЖТ
   270тЖТ   270тЖТdef _extract_referral(message: Message) -> int | None:
   271тЖТ   271тЖТ    """Extract referral parameter from /start command deep link."""
   272тЖТ   272тЖТ    if message.text and message.text.startswith("/start ref_"):
   273тЖТ   273тЖТ        try:
   274тЖТ   274тЖТ            return int(message.text.split("ref_")[1])
   275тЖТ   275тЖТ        except (ValueError, IndexError):
   276тЖТ   276тЖТ            return None
   277тЖТ   277тЖТ    return None
   278тЖТ   278тЖТ```
   279тЖТ   279тЖТ
   280тЖТ   280тЖТ```python
   281тЖТ   281тЖТ# adapters/telegram/presenters/user_presenter.py
   282тЖТ   282тЖТclass UserPresenter:
   283тЖТ   283тЖТ    """Presenter: translates use case output into Telegram-friendly text."""
   284тЖТ   284тЖТ
   285тЖТ   285тЖТ    def format_welcome(self, response) -> str:
   286тЖТ   286тЖТ        if response.is_new_user:
   287тЖТ   287тЖТ            return (
   288тЖТ   288тЖТ                f"Welcome, {response.username}! ЁЯОЙ\n"
   289тЖТ   289тЖТ                f"Your account has been created.\n"

   303тЖТ   303тЖТ### Dependency Injection via aiogram Middleware
   304тЖТ   304тЖТ
   305тЖТ   305тЖТaiogram's middleware system provides a natural mechanism for injecting use case dependencies into handlers:
   306тЖТ   306тЖТ
   307тЖТ   307тЖТ```python
   308тЖТ   308тЖТ# infrastructure/middleware/use_case_middleware.py
   309тЖТ   309тЖТfrom aiogram import BaseMiddleware
   310тЖТ   310тЖТfrom typing import Any, Callable, Awaitable
   311тЖТ   311тЖТ
   312тЖТ   312тЖТfrom application.use_cases.register_user import RegisterUserUseCase
   313тЖТ   313тЖТ
   314тЖТ   314тЖТ
   315тЖТ   315тЖТclass UseCaseMiddleware(BaseMiddleware):
   316тЖТ   316тЖТ    """Injects use case instances into handler data dictionary."""
   317тЖТ   317тЖТ
   318тЖТ   318тЖТ    def __init__(self, register_user: RegisterUserUseCase) -> None:
   319тЖТ   319тЖТ        self._register_user = register_user
   320тЖТ   320тЖТ        super().__init__()
   321тЖТ   321тЖТ
   322тЖТ   322тЖТ    async def __call__(
   323тЖТ   323тЖТ        self,
   324тЖТ   324тЖТ        handler: Callable[[Any, dict[str, Any]], Awaitable[Any]],
   325тЖТ   325тЖТ        event: Any,
   326тЖТ   326тЖТ        data: dict[str, Any],
   327тЖТ   327тЖТ    ) -> Any:
   328тЖТ   328тЖТ        data["register_user"] = self._register_user
   329тЖТ   329тЖТ        return await handler(event, data)
   330тЖТ   330тЖТ```
   331тЖТ   331тЖТ
   332тЖТ   332тЖТ```python
   333тЖТ   333тЖТ# infrastructure/main.py тАФ wiring everything together
   334тЖТ   334тЖТfrom aiogram import Bot, Dispatcher
   335тЖТ   335тЖТfrom aiogram.fsm.storage.redis import RedisStorage
   336тЖТ   336тЖТfrom redis.asyncio import Redis
   337тЖТ   337тЖТ
   338тЖТ   338тЖТfrom infrastructure.config import Settings
   339тЖТ   339тЖТfrom infrastructure.database.session import create_session_factory
   340тЖТ   340тЖТfrom adapters.database.user_repository import SQLAlchemyUserRepository
   341тЖТ   341тЖТfrom adapters.llm.openai_client import OpenAILLMService
   342тЖТ   342тЖТfrom application.use_cases.register_user import RegisterUserUseCase
   343тЖТ   343тЖТfrom adapters.telegram.user_controller import router as user_router
   344тЖТ   344тЖТ
   345тЖТ   345тЖТ
   346тЖТ   346тЖТasync def create_app() -> Dispatcher:
   347тЖТ   347тЖТ    settings = Settings()
   348тЖТ   348тЖТ
   349тЖТ   349тЖТ    # Framework layer setup
   350тЖТ   350тЖТ    redis = Redis.from_url(settings.redis_url)
   351тЖТ   351тЖТ    storage = RedisStorage(redis=redis)
   352тЖТ   352тЖТ    bot = Bot(token=settings.bot_token)
   353тЖТ   353тЖТ    dp = Dispatcher(storage=storage)
   354тЖТ   354тЖТ
   355тЖТ   355тЖТ    # Infrastructure layer тАФ create session factory
   356тЖТ   356тЖТ    session_factory = create_session_factory(settings.database_url)
   357тЖТ   357тЖТ
   358тЖТ   358тЖТ    # Adapter implementations
   359тЖТ   359тЖТ    user_repo = SQLAlchemyUserRepository(session_factory)
   360тЖТ   360тЖТ
   361тЖТ   361тЖТ    # Use case construction тАФ depends on abstract ports, gets concrete adapters
   362тЖТ   362тЖТ    register_user = RegisterUserUseCase(
   363тЖТ   363тЖТ        user_repo=user_repo,
   364тЖТ   364тЖТ        referral_service=...,  # another adapter
   365тЖТ   365тЖТ    )
   366тЖТ   366тЖТ
   367тЖТ   367тЖТ    # Wire middleware for DI
   368тЖТ   368тЖТ    dp.message.middleware(UseCaseMiddleware(register_user=register_user))
   369тЖТ   369тЖТ
   370тЖТ   370тЖТ    # Register routers (controllers)
   371тЖТ   371тЖТ    dp.include_router(user_router)
   372тЖТ   372тЖТ
   373тЖТ   373тЖТ    return dp
   374тЖТ   374тЖТ```

   654тЖТ   654тЖТ## Migration Guide from "Big Ball of Mud"
   655тЖТ   655тЖТ
   656тЖТ   656тЖТMany Telegram bots start as a single Python file with all logic crammed into aiogram handlers. Here is a step-by-step migration path:
   657тЖТ   657тЖТ
   658тЖТ   658тЖТ### Step 1: Identify and Extract Domain Logic
   659тЖТ   659тЖТ
   660тЖТ   660тЖТFind business rules buried in handlers and extract them into entity classes.
   661тЖТ   661тЖТ
   662тЖТ   662тЖТ**Before (all in handler):**
   663тЖТ   663тЖТ```python
   664тЖТ   664тЖТ@router.message(Command("ban"))
   665тЖТ   665тЖТasync def ban_user(message: Message, session: AsyncSession):
   666тЖТ   666тЖТ    target_id = int(message.text.split()[1])
   667тЖТ   667тЖТ    result = await session.execute(select(DBUser).where(DBUser.id == target_id))
   668тЖТ   668тЖТ    user = result.scalar()
   669тЖТ   669тЖТ    if user.role == "admin":
   670тЖТ   670тЖТ        await message.answer("Cannot ban an admin!")
   671тЖТ   671тЖТ        return
   672тЖТ   672тЖТ    user.is_active = False
   673тЖТ   673тЖТ    await session.commit()
   674тЖТ   674тЖТ    await message.answer("User banned.")
   675тЖТ   675тЖТ```
   676тЖТ   676тЖТ
   677тЖТ   677тЖТ**After (domain logic in entity):**
   678тЖТ   678тЖТ```python
   679тЖТ   679тЖТ# domain/entities/user.py
   680тЖТ   680тЖТ@dataclass
   681тЖТ   681тЖТclass User:
   682тЖТ   682тЖТ    # ... fields ...
   683тЖТ   683тЖТ    def ban(self, banner_role: UserRole) -> None:
   684тЖТ   684тЖТ        if self.role == UserRole.ADMIN:
   685тЖТ   685тЖТ            raise PermissionError("Cannot ban an admin")
   686тЖТ   686тЖТ        if not self.is_active:
   687тЖТ   687тЖТ            raise ValueError("User is already banned")
   688тЖТ   688тЖТ        self.is_active = False
   689тЖТ   689тЖТ
   690тЖТ   690тЖТ# handler now delegates to domain
   691тЖТ   691тЖТ@router.message(Command("ban"))
   692тЖТ   692тЖТasync def ban_user(message: Message, ban_user_use_case: BanUserUseCase):
   693тЖТ   693тЖТ    try:
   694тЖТ   694тЖТ        response = await ban_user_use_case.execute(BanUserRequest(
   695тЖТ   695тЖТ            target_id=int(message.text.split()[1]),
   696тЖТ   696тЖТ            banner_telegram_id=message.from_user.id,
   697тЖТ   697тЖТ        ))
   698тЖТ   698тЖТ        await message.answer("User banned." if response.success else "Ban failed.")
   699тЖТ   699тЖТ    except PermissionError:
   700тЖТ   700тЖТ        await message.answer("Cannot ban an admin!")
   701тЖТ   701тЖТ```
   702тЖТ   702тЖТ
   703тЖТ   703тЖТ### Step 2: Define Use Case Interfaces
   704тЖТ   704тЖТ
   705тЖТ   705тЖТCreate use case classes with explicit input/output DTOs. Each handler should call exactly one use case.
   706тЖТ   706тЖТ
   707тЖТ   707тЖТ### Step 3: Introduce Repository Abstractions
   708тЖТ   708тЖТ
   709тЖТ   709тЖТReplace direct SQLAlchemy queries in handlers with repository interfaces. Start with the interface in the use case, then implement it with SQLAlchemy in the adapters layer.
   710тЖТ   710тЖТ
   711тЖТ   711тЖТ### Step 4: Move Handlers to Adapter Layer
   712тЖТ   712тЖТ
   713тЖТ   713тЖТHandlers become thin controllers that:
   714тЖТ   714тЖТ1. Parse the aiogram `Message`/`CallbackQuery` into a request DTO
   715тЖТ   715тЖТ2. Call the use case
   716тЖТ   716тЖТ3. Format the response using a presenter
   717тЖТ   717тЖТ4. Send the reply via aiogram
   718тЖТ   718тЖТ
   719тЖТ   719тЖТ### Step 5: Wire Dependencies
   720тЖТ   720тЖТ
   721тЖТ   721тЖТUse aiogram middleware or a DI container to inject use case instances into handlers. The composition root (your `main.py`) is the only place where concrete implementations are coupled to abstract interfaces.
   722тЖТ   722тЖТ

   725тЖТ   725тЖТ## Common Pitfalls
   726тЖТ   726тЖТ
   727тЖТ   727тЖТ### Pitfall 1: Leaking aiogram Types into Use Cases
   728тЖТ   728тЖТ
   729тЖТ   729тЖТNever pass `aiogram.types.Message` into a use case. The use case should accept a plain dataclass DTO. If you find yourself importing from `aiogram` in your `application/` directory, you have violated the Dependency Rule.
   730тЖТ   730тЖТ
   731тЖТ   731тЖТ### Pitfall 2: Anemic Domain Models
   732тЖТ   732тЖТ
   733тЖТ   733тЖТEntities with only data fields and no behavior are "anemic." Push business rules into entities тАФ a `User` entity should enforce its own invariants, not rely on handlers or use cases to validate constraints.
   734тЖТ   734тЖТ
   735тЖТ   735тЖТ### Pitfall 3: Over-Engineering Small Bots
   736тЖТ   736тЖТ
   737тЖТ   737тЖТClean Architecture adds structural overhead. For a simple notification bot with three commands, a single-file approach may be more appropriate. Apply Clean Architecture when your bot has:
   738тЖТ   738тЖТ- Multiple bounded domains (chat, admin, subscriptions, AI)
   739тЖТ   739тЖТ- Complex business rules that need testing in isolation
   740тЖТ   740тЖТ- More than one developer working on the codebase
   741тЖТ   741тЖТ- A need to swap infrastructure (e.g., switch from OpenAI to Anthropic)
   742тЖТ   742тЖТ
   743тЖТ   743тЖТ### Pitfall 4: Skipping the Presenter
   744тЖТ   744тЖТ
   745тЖТ   745тЖТResist the temptation to format Telegram messages directly in handlers. A presenter separates "what to say" from "how to say it," enabling you to change message formatting without touching use case logic.
   746тЖТ   746тЖТ
   747тЖТ   747тЖТ### Pitfall 5: God Use Cases
   748тЖТ   748тЖТ
   749тЖТ   749тЖТA use case should represent one user intention. If your `ProcessMessage` use case is 300 lines long, it is doing too much. Decompose it into smaller, composable use cases or extract domain services.
   750тЖТ   750тЖТ
   751тЖТ   751тЖТ---
   752тЖТ   752тЖТ
   753тЖТ   753тЖТ## Best Practices
   754тЖТ   754тЖТ
   755тЖТ   755тЖТ1. **Start with the domain.** Write entities and use cases before writing any aiogram code. This forces you to think about what the bot does, not how it communicates.
   756тЖТ   756тЖТ
   757тЖТ   757тЖТ2. **Use Protocol or ABC for ports.** Python's `typing.Protocol` (structural subtyping) or `abc.ABC` (nominal subtyping) both work. `Protocol` is more Pythonic and allows duck-typed implementations.
   758тЖТ   758тЖТ
   759тЖТ   759тЖТ3. **Keep DTOs flat and simple.** Request and response DTOs should be plain dataclasses or Pydantic models with no methods. They are data carriers, not behavior containers.
   760тЖТ   760тЖТ
   761тЖТ   761тЖТ4. **One use case per file.** Each use case is a self-contained unit with its own request/response types and port definitions. This makes the codebase navigable.
   762тЖТ   762тЖТ
   763тЖТ   763тЖТ5. **Dependency injection at the composition root.** The `main.py` module is the only place that knows about concrete implementations. Everywhere else depends on abstractions.
   764тЖТ   764тЖТ
   765тЖТ   765тЖТ6. **Test from the inside out.** Start with entity unit tests, then use case tests with mocked ports, then adapter integration tests. The further out you go, the fewer tests you need.
   766тЖТ   766тЖТ
   767тЖТ   767тЖТ7. **Use dataclasses for entities, Pydantic for DTOs.** Entities are internal domain objects тАФ dataclasses keep them lightweight. DTOs cross boundaries and benefit from Pydantic's validation.
   768тЖТ   768тЖТ
   769тЖТ   769тЖТ8. **Named constructors for complex creation.** Use `@classmethod` factory methods for entity construction that requires validation or defaults.
   770тЖТ   770тЖТ

   773тЖТ   773тЖТ## Related Patterns
   774тЖТ   774тЖТ
   775тЖТ   775тЖТ- **Hexagonal Architecture (Ports & Adapters):** A related pattern that emphasizes explicit ports and adapters rather than concentric layers. See [Hexagonal Architecture](../hexagonal/README.md).
   776тЖТ   776тЖТ- **Domain-Driven Design:** Provides tactical patterns (aggregates, value objects, domain events) that populate the entity layer. See [DDD](../ddd/README.md).
   777тЖТ   777тЖТ- **CQRS:** Separates read and write operations, often layered on top of Clean Architecture's use case pattern. See [CQRS](../cqrs/README.md).
   778тЖТ   778тЖТ- **Event-Driven Architecture:** Domain events provide a decoupled way for use cases to trigger side effects. See [Event-Driven](../event-driven/README.md).
